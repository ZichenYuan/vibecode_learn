# Note Template

This is the reference template for learning notes generated by the learn-mode skill. Every note saved to `~/learning-notes/` should follow this format.

Notes use YAML frontmatter for compatibility with **Obsidian**, **Notion**, and other knowledge tools. The body is structured for progressive detail — scan the TL;DR, or read deeper as needed.

---

## Template

```markdown
---
title: "{Topic Title}"
date: {YYYY-MM-DD}
category: "{category}"
tags: ["{tag1}", "{tag2}", "{tag3}"]
difficulty: "{beginner | intermediate | advanced}"
status: "{seedling | growing | evergreen}"
---

# {Topic Title}

> **TL;DR:** {One sentence — the core idea you'd tell a colleague in 10 seconds.}

**Context:** {What task triggered this learning — 1 sentence}

---

## What Is It?

{1-2 paragraph plain-English explanation. Write as if explaining to a smart friend who hasn't encountered this before. Use analogies where helpful.}

## Code Example

{Minimal, runnable code example. Include comments on non-obvious lines.}

```{language}
// Example code here
```

## Why It Matters

{When would you use this? What problem does it solve?}

- {Practical reason 1}
- {Practical reason 2}

## Common Gotchas

1. **{Gotcha title}** — {Explanation and how to avoid it}
2. **{Gotcha title}** — {Explanation and how to avoid it}

## Key Takeaways

- {Takeaway 1}
- {Takeaway 2}
- {Takeaway 3}

## Related

- [[{Related note title}]] — {why it's related, 3-5 words}
- [{Official docs link title}]({url})

---

*Generated by learn-mode skill*
```

## Frontmatter Fields

### Required
| Field | Purpose | Example |
|-------|---------|---------|
| `title` | Note title, matches the H1 | `"Python List Comprehensions"` |
| `date` | When the note was created | `2026-03-03` |
| `category` | Primary topic category from SKILL.md 3.1 | `"language"` |
| `tags` | 2-4 searchable keywords — concepts, languages, tools | `["python", "loops", "syntax"]` |

### Optional
| Field | Purpose | Values |
|-------|---------|--------|
| `difficulty` | How advanced the concept is | `beginner`, `intermediate`, `advanced` |
| `status` | Note maturity — update as understanding deepens | `seedling` (just learned), `growing` (revisited), `evergreen` (solid) |

**Why frontmatter?** Obsidian uses it for graph view, Dataview queries, and search. Notion imports it as properties. Plain markdown readers ignore it gracefully.

## Guidelines

### Length
- Aim for **50-150 lines** per note
- Be concise — this is a reference, not a textbook
- If a topic is large, split into focused sub-notes

### The TL;DR Rule
Every note must start with a one-sentence TL;DR right after the frontmatter. This is the **30-second revisit** — if you only have time to glance at the note, this line should be enough to jog your memory.

### Code Examples
- Keep examples **minimal** — shortest code that demonstrates the concept
- Always include **comments** on non-obvious lines
- Only show **correct** code — don't include broken or incorrect examples
- Use the actual language from the session (don't translate to a different language)

### Linking
- Use **wikilinks** `[[Note Title]]` as the primary link style (works in Obsidian, converts cleanly to Notion)
- Also include a relative path fallback: `[[Error Handling]](../errors/error-handling.md)` for plain markdown readers
- Link to official docs when available
- After accumulating several notes, cross-reference them

### Tags Strategy
Pick tags that answer: "What would I search for to find this note later?"
- Include the **language** if applicable (`python`, `go`, `javascript`)
- Include the **concept** (`recursion`, `async`, `error-handling`)
- Include the **context** if distinctive (`api`, `cli`, `frontend`)
- Keep tags lowercase, hyphenated for multi-word (`error-handling`, not `Error Handling`)

### Updating Existing Notes
When appending to an existing note, add a new section and update the frontmatter:

```markdown
---

## Update: {YYYY-MM-DD}

**Context:** {What new thing was learned}

{New content following the same structure}
```

When updating, also:
- Bump `status` if understanding has deepened (e.g., `seedling` → `growing`)
- Add any new `tags` to the frontmatter
- Update the TL;DR if the core understanding has changed

This preserves the learning timeline while keeping everything in one place.
